# Auth

Three identity concepts, three token types, one threat model.

## Identity concepts

**User identity** — a specific human employee. Used for audit log attribution, RBAC decisions, and contribution attribution.

**Agent identity** — the LLM client acting on behalf of a user (Claude Desktop, Cursor, a scheduled agent). Agents don't have their own identity; they inherit the user's identity via the user's token.

**Service identity** — a backend component (ingestion worker, Slack bot) acting on behalf of the organization rather than a specific user.

## Token types

### Per-user tokens (the common case)

Issued to individual employees during onboarding. Used by their MCP client, the Web UI (as an alternative to the OIDC cookie), and any personal automation.

- Generated at onboarding via the Web UI's self-service page after OIDC login
- Signed JWT with claims: `sub` (user id), `roles` (see RBAC below), `iat`, `exp`, `jti`
- **Default lifetime:** 90 days, refreshable
- **Revocation:** `jti` checked against a revocation list in Postgres on every request (cached in memory for 60s)
- Stored by the client: MCP client config env var, Web UI localStorage with short idle timeout

### Service tokens

Issued to backend components. Used for server-to-server calls that aren't tied to a specific user.

- Generated by admins via the Web UI admin panel
- Signed JWT with claims: `sub` (service name), `roles: ["service"]`, no expiry
- Rotated on a schedule (every 180 days recommended)
- **Never** used by a human client
- Audit log records the service name; for writes the service must also pass a `on_behalf_of` user id header

### OIDC session cookies (Web UI only)

Managed by the OIDC provider (Clerk/Auth0/Descope/company SSO).

- Short-lived (30 min idle, 8 hour hard expiry)
- HttpOnly, Secure, SameSite=Lax
- Refresh tokens rotate per provider default

## Provider recommendations

| OIDC provider | Use when | Notes |
|---|---|---|
| **Clerk** | Smaller teams, fast setup | Free tier generous, best DX |
| **Auth0** | Mid-size, common enterprise pick | Paid quickly; solid feature set |
| **Descope** | Want passwordless / passkey-first | Marketplace integration on Vercel |
| **WorkOS** | Selling into enterprises that demand SAML | Best SSO integration depth |
| **Company SSO (Okta/Entra/Google)** | Existing enterprise | Skip the third-party, integrate direct |

## RBAC

Three roles. Additive: `admin > curator > reader`.

| Role | Can read | Can propose | Can approve | Can delete | Can manage users |
|---|---|---|---|---|---|
| `reader` | yes | no | no | no | no |
| `curator` | yes | yes | yes | own entries only | no |
| `admin` | yes | yes | yes | any | yes |

- Every authenticated user is at minimum a `reader`
- Promoting to `curator` happens in the Web UI admin panel; records an audit event
- `admin` is a bootstrap role: the first user to log in becomes admin; subsequent admins are promoted
- **Optional scope-level RBAC (v2):** curators scoped to a subset of primitive types or a tag prefix. Not in v1.

## Threat model

### What we protect against

| Threat | Control |
|---|---|
| Unauthenticated access to the Context Store | All reads and writes require a valid token |
| Stolen per-user token | 90-day expiry, revocation list, ability to revoke from Web UI |
| Privilege escalation | Role changes are audit-logged, only admins can promote |
| Unauthorized write / poisoning the store | Writes land in review queue; never auto-publish |
| Data exfil via MCP | Audit log captures every read; reads are rate-limited per user |
| Rogue ingestion worker scraping things it shouldn't | Service tokens scoped to specific ingestion sources, audit trail |

### What we don't protect against (and should be honest about)

| Non-threat | Why we don't |
|---|---|
| A determined admin exfiltrating the entire store | Admins have read access by design |
| A user's LLM client (Claude Desktop) storing query results locally | Out of scope — that's the client's trust domain |
| Social engineering that gets a curator to approve a malicious entry | Process problem, not auth problem — addressed by review-queue UX nudges |
| Prompt injection through ingested content causing bad proposals | See [`INGESTION.md`](./INGESTION.md) — proposals always go through human review |

## Network posture

- MCP server: **internal DNS only.** `lexicon-mcp.company.internal`, reachable from the corp VPN or the office network, not the public internet. Per-user token is defense in depth, not the only layer.
- Web UI: public DNS + OIDC-gated. `lexicon.company.com` is fine.
- REST API: same DNS as the Web UI; uses OIDC cookie or per-user token.
- Postgres: never publicly reachable. Allowlist from MCP server, REST API, and ingestion workers only.

## Onboarding flow

1. New employee joins the company, gets SSO access
2. They navigate to `lexicon.company.com`, sign in via SSO
3. First-visit flow shows a one-page explanation of what the Context Store is
4. A button: "Get my MCP token"
5. Page displays a token (once) and a copy-paste block of MCP client configs for Claude Desktop / Cursor / Windsurf / etc.
6. Employee pastes the config into their client, restarts it, asks Claude a test question
7. Token is stored in a hashed form in the DB for revocation; plaintext is never persisted server-side after initial display

## Offboarding flow

1. When a user is deactivated in SSO, the Context Store listens for the SCIM deprovision event (or polls nightly)
2. All per-user tokens for that user are revoked (their `jti`s land on the revocation list)
3. Their proposals/comments remain in the audit log under their (now-inactive) user id, for historical integrity
4. Admin can reassign ownership of any entries where `owner` was the deactivated user

## Audit log linkage

Every tool call, every write, every role change, every token issuance is logged with:
- `user_id` (or service name for service tokens)
- `agent_hint` (from MCP client identification, best-effort)
- `tool` or `action`
- `args` (sanitized — PII fields masked per [`GOVERNANCE.md`](./GOVERNANCE.md))
- `result_id` or affected-row pointer
- `ip_address`, `user_agent`
- `ts`

Audit log is append-only. Admins can query it via the Web UI; exported to SIEM via a log shipper.

## Rotation procedures

**Per-user tokens:** users rotate from the Web UI ("revoke and reissue") or automatically after 90 days via email reminder.

**Service tokens:** admins rotate via Web UI. Old token remains valid for a 24-hour overlap window while services pick up the new one.

**HMAC signing secret (`LEXICON_SERVICE_TOKEN_HMAC_SECRET`):** rotate yearly. Dual-verify during rotation: accept tokens signed with either old or new secret for 48 hours.

## What not to do

- Don't skip per-user tokens for "convenience." The audit log is worthless without user attribution.
- Don't allow MCP reads without auth even on the internal network. A compromised laptop in the office should not be able to exfiltrate the store.
- Don't let service tokens become long-lived super-credentials; rotate them on schedule and scope them narrowly.
- Don't store tokens in git, Slack, or anywhere outside the provider's secret store.
